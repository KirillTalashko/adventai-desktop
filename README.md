# AdventAI Desktop — Визовый специалист

Десктопное приложение для Windows на **Compose for Desktop (Kotlin)** — AI-агент «Визовый специалист»
с явной моделью памяти. Часть **AI Advent Challenge #8**.

> Открывается и запускается в **IntelliJ IDEA 2025.3** (Gradle-проект): `File → Open` → папка проекта →
> Gradle sync → задача `run`. Сборка в самодостаточный `.exe` — `gradlew createDistributable`.

## День 11 — «Модель памяти ассистента»

**Задание дня:** описать и реализовать модель памяти ассистента — разделить информацию минимум на 3 типа
(краткосрочная / рабочая / долговременная), хранить типы раздельно, явно выбирать, что и куда сохраняется,
и показать, как это влияет на ответы.

Здесь модель памяти не только реализована, но и **наполняется и управляется автоматически** — отдельными
служебными агентами под капотом.

### Что реализовано

**1. Три слоя памяти (раздельное хранение)**

| Слой | Что хранит | Scope | Где хранится |
|---|---|---|---|
| **Краткосрочная** | реплики текущего диалога | диалог | `conversations/<id>.json` |
| **Рабочая** | цель и ограничения текущей задачи | диалог | `working/<id>.json` (структурно) |
| **Долговременная** | профиль пользователя + принятые решения | агент, между сессиями | `memory/profile.md` + `memory/decisions.md` |

**2. Авто-наполнение памяти (фоновый агент)** — `MemoryExtractor`
После каждого ответа отдельный служебный агент (всегда **`deepseek-chat`** для стабильности) читает
последние реплики + снимок текущей памяти, извлекает **только новые** факты и раскладывает их по слоям
(`ЦЕЛЬ`/`ОГРАНИЧЕНИЯ` → рабочая; `ПРОФИЛЬ`/`РЕШЕНИЯ` → долговременная). Работает в фоне, не блокирует UI,
не выдумывает (строго из текста), с дедупликацией. Вручную дополнять память тоже можно — окно «Память».

**3. Управление контекстом — под капотом, без режимов** — `ContextAssembler`
Каждый запрос собирается единообразно:
```
system( базовый промпт + [ДОЛГОВРЕМЕННАЯ] + [РАБОЧАЯ] [+ [РЕЗЮМЕ]] ) + последние N = 12 реплик
```
- пока окно свободно (< 30%) — уходит вся история;
- при заполнении **≥ 30%** старый хвост (старше последних 12 реплик) автоматически сворачивается в
  rolling summary. Смысл не теряется, потому что важное уже живёт в долговременной/рабочей памяти.

**4. Видно влияние на ответы**
- окно **«Память»** показывает текущее содержимое всех слоёв (видно, как они растут по ходу диалога);
- факты из памяти инжектируются в system prompt → агент «помнит» пользователя и задачу;
- под каждым ответом — расход токенов, в композере — сумма по сессии и стоимость.

### Где какая техника применяется (всё под капотом)

| Техника | Слой | Роль |
|---|---|---|
| **Sliding window** | краткосрочная | последние 12 реплик дословно |
| **Sticky facts** | рабочая (+ промоушен в долговременную) | факты задачи «прибиты», не выпадают при обрезке |
| **Summary** | переполнение окна | свёртка старого хвоста при ≥ 30% |
| **Branching** | — | альтернативные ветки диалога (в роадмапе) |

## День 12 — «Персонализация»

**Задание дня:** научить агента подстраиваться под конкретного пользователя — хранить его предпочтения
и учитывать их в каждом ответе.

Ключевое различие, на котором строится день: **профиль = КАК отвечать** (явные предпочтения, задаёт
пользователь) против **память = ЧТО известно** (факты, агент извлекает сам, см. День 11).

### Что реализовано

**1. Профиль предпочтений** — `UserProfile`
Длина ответа · тон · формат (чек-листы / пошагово / без воды / с примерами) · ограничения · язык.
Подмешивается в **system prompt** каждого запроса отдельным блоком `[ПРОФИЛЬ ПОЛЬЗОВАТЕЛЯ — как отвечать]`
рядом с блоками памяти — стиль и факты не смешиваются.

**2. Онбординг** при первом запуске — заполнение профиля до начала работы.

**3. Локальные аккаунты с изоляцией данных** — `AccountStore`
У каждого аккаунта свои диалоги, память и профиль в `accounts/<id>/`. Ключи API — глобальные (общие).
Переключение и создание аккаунтов — из меню профиля в сайдбаре.

**4. Профиль → долговременная память (один слой фактов)**
Факты из профиля (имя, описание) при онбординге **засеваются** в долговременную память, а при
редактировании профиля **синхронизируются** (diff: убрать изменившийся старый факт, добавить новый),
не затрагивая факты, добавленные `MemoryExtractor` из диалога. «ЧТО известно» живёт в одном слое.

**5. Выход из аккаунта** — `logout()`
Выход не удаляет данные: забывается активный аккаунт и показывается экран **«С возвращением»** со списком
сохранённых аккаунтов для входа обратно (или создания нового). Выход переживает перезапуск приложения.
Аккаунт можно **удалить** (с подтверждением) — вместе со всеми его данными на диске.

## День 13 — «Состояние задачи (Task State Machine)»

**Задание дня:** формализовать состояние задачи как конечный автомат — этап, текущий шаг, ожидаемое
действие; проверить паузу на любом этапе и продолжение без повторных объяснений.

Здесь это не просто FSM, а **многоагентный оркестратор как у Claude Code**: на каждую стадию — свой агент
со своим system prompt и отдельным вызовом API, переходы разрешает **код, а не LLM**.

### Конечный автомат

`INTAKE → PLANNING → EXECUTION → VALIDATION → DONE`. Состояние — это **данные** (`TaskContext`): этап,
план, текущий шаг, что сделано, документы, ожидаемое действие. Переходы — по карте разрешённых
(`transitionTo`), нелегальный переход бросает исключение. Состояние лежит в `conversations/<id>.json`,
поэтому **пауза/возобновление бесплатны** (переживают перезапуск), а блок `[STATE]` инжектится агенту →
**продолжение без повторных объяснений**.

### Агент на каждую стадию (`TaskOrchestrator`)

| Стадия | Агент | Роль |
|---|---|---|
| INTAKE | интервьюер | уточняет недостающие вводные (или сразу отвечает на простой вопрос) |
| PLANNING | планировщик | предлагает **4 подхода** (выбор в чате + «Другое»), затем строит план |
| EXECUTION | исполнитель | выполняет **ровно текущий** шаг; нужный документ не блокирует, уходит в «приложить позже» |
| VALIDATION | валидатор | проверяет работу; недостающие файлы — не повод заваливать, лимит доработок |
| — | ассистент | отвечает на вопросы пользователя по ходу задачи, **не сдвигая** автомат |

### Взаимодействие (как Claude Code, без панелей)

- Никаких таблиц сверху/снизу — всё в потоке диалога. Снизу **статус-строка**: эмблемка · этап
  (думает/планирует/исполняет/проверяет) · время · токены.
- Служебные переходы — автоматически; **план показывается чекпоинтом**, шаги подтверждаются по одному
  («Дальше — шаг N: …»).
- Кнопка **«+»** — приложить Word/PDF, когда агент просит документ; «приложу позже» запоминается в памяти.
- В финале честно: если документы не приложены — **«Почти готово, осталось приложить…»** с кнопками, а не
  глухое «готово».

Подробности и карта файлов — [`.claude/STATE_MACHINE.md`](.claude/STATE_MACHINE.md).

## День 14 — «Инварианты и ограничения состояния»

**Задание дня:** добавить ассистенту инварианты, которые он не имеет права нарушать (архитектура,
техрешения, ограничения по стеку, бизнес-правила); хранить их отдельно от диалога, явно учитывать в
рассуждениях и отказываться от решений, которые их нарушают.

### Что реализовано (двойная защита)

**1. Модель и хранение отдельно от диалога** — `Invariant`
Правило на естественном языке. Встроенные жёсткие правила консультанта — в коде (`BUILT_IN_INVARIANTS`:
только законные пути; без юр-гарантий; оставаться в роли). Пользовательские — в `accounts/<id>/invariants.json`.

**2. Инжект в промпт (1-й рубеж)** — `renderInvariantsBlock`
Блок `[ИНВАРИАНТЫ — нельзя нарушать]` подмешивается в системный промпт **чата** (`ContextAssembler`) и
**стадийных агентов задачи** (`TaskOrchestrator`). Ассистент учитывает их в рассуждениях и при конфликте
сам отказывается, называя нарушаемый инвариант.

**3. Страж — пост-проверка (2-й рубеж)** — `InvariantGuard`
После ответа отдельный служебный агент (deepseek) проверяет его на нарушение активных инвариантов
(`VIOLATION: …` / `OK`). При нарушении ответ **перегенерируется в обоснованный отказ**. Работает на
пользовательских стадиях оркестратора (intake / выполнение шага / ответ-ассистент).

**4. Поведение при конфликте**
Запрос противоречит инварианту → ассистент не предлагает нарушающее решение, прямо отказывается, называет
правило и кратко объясняет причину, предлагает допустимую альтернативу.

**5. UI** — окно **«Инварианты»** (кнопка в сайдбаре): встроенные (просмотр) + свои (добавить / вкл-выкл /
удалить).

Также поправлена **память** (баги Дня 11, всплывшие при тестах): экстрактор больше не пишет плейсхолдеры
(`<нет новой информации>`) и не выдумывает цели — берёт факты только из слов пользователя; профиль/память
инжектятся как фон, а не как «железный» факт. Подробности — [`.claude/INVARIANTS.md`](.claude/INVARIANTS.md).

## День 15 — «Контролируемые переходы состояний»

**Задание дня:** явные переходы между состояниями задачи — допустимые состояния, разрешённые переходы,
ассистент не может «перепрыгнуть» этап (нельзя реализация до плана, нельзя финал без валидации).

Ядро заложено в Дне 13 (`TaskState` + `TRANSITIONS` + `transitionTo`, «переходы решает код, а не LLM»);
в Дне 15 это **усилено и сделано демонстрируемым**:
- **Безопасный API переходов** — `canTransition` / `tryTransition` / `allowedNext`; `transitionTo` бросает
  понятную ошибку «этапы нельзя перепрыгивать».
- **Структурная гарантия:** FSM двигает только код. `EXECUTION` достижим лишь из `PLANNING` (после готового
  плана), `DONE` — лишь из `VALIDATION`. LLM состоянием не управляет (антипаттерн «решения за LLM в тексте»).
- **Реакция на попытку пропустить этап:** жизненный цикл описан в `[STATE]`, и на просьбу «сразу финал /
  без проверки» ассистент объясняет, что перепрыгивать нельзя, и называет ближайший допустимый шаг
  (само правило — в коде, промпт лишь объясняет отказ).
- **Пауза/продолжение:** состояние на диске, цикл продолжается с того же легального этапа.

**Рой агентов (итог недели).** Дни 11–15 соединены: память (11) · персонализация (12) · автомат задачи (13)
· инварианты (14) · контролируемые переходы (15). Специализированные роли: интервьюер, планировщик
(варианты + план), исполнитель, валидатор, ассистент по задаче, `MemoryExtractor`, `InvariantGuard`.

**Бонус — проактивный помощник.** Новый агент-«разведчик возможностей» (`OfferAgent`) после релевантного
шага **предлагает** (с правом отказаться) пройти **пробное собеседование**. Согласие открывает отдельное
окно, где «визовый офицер» (`MockInterviewAgent`) ведёт интервью и даёт оценку готовности; основная задача
при этом не двигается — после окончания возвращаемся на свой шаг, итог фиксируется в диалоге.

Подробности — [`.claude/STATE_MACHINE.md`](.claude/STATE_MACHINE.md).

## День 16 — «Подключение MCP»

**Задание дня:** установить MCP SDK / клиент (или поднять MCP-сервер для локального варианта); минимальным
кодом установить **MCP-соединение** и получить **список доступных инструментов**; проверить, что соединение
устанавливается и список возвращается корректно.

Здесь это не только консольный пример, но и **живая демонстрация в приложении**: агент подключается к
локальному MCP-серверу и показывает его инструменты прямо в окне.

### Что реализовано

**1. Доменный порт инструментов** — `ToolGateway` (+ модель `Tool`)
`connect` / `listTools` / `callTool` / `close`. Домен знает про «инструменты», но не про SDK, процессы и
JSON-RPC — границы Clean Architecture сохранены (по образцу `LlmGateway`).

**2. MCP-клиент** — `McpClient` (data)
На официальном **MCP Kotlin SDK**. Поднимает локальный сервер **отдельным процессом** и общается с ним по
**stdio** (JSON-RPC): `initialize` → `tools/list` → `tools/call`. Ответы маппятся в доменный `Tool`.

**3. Свой минимальный MCP-сервер** — `VisaMcpServer`
`Server` + `StdioServerTransport` с двумя инструментами: `ping` (→ «pong», проверка связи) и заглушка
`get_visa_news(country)` (реальный парсинг официальных сайтов — День 17). Протокол идёт по stdout, поэтому
логи заглушены (`slf4j-nop`).

**4. Проверка — консоль и UI**
- `gradlew runMcpDemo` — подключается и печатает список инструментов;
- в приложении кнопка-пазл рядом с «+» открывает окно **«Инструменты MCP»**: статус соединения, список
  инструментов со схемами и кнопка **«Проверить связь (ping)»** → `→ pong` (живой запрос-ответ).

| Проверка | Как | Результат |
|---|---|---|
| Соединение устанавливается | клик по кнопке / `runMcpDemo` | «Соединение установлено» |
| Список инструментов | `tools/list` | `ping`, `get_visa_news` (+ схемы) |
| Вызов инструмента | «Проверить связь (ping)» | `→ pong` |
| Сервер — реальный процесс | `Get-CimInstance … VisaMcpServerKt` | PID живёт, пока окно открыто |

**Заметка по версии.** MCP Kotlin SDK зафиксирован на **0.10.0** — последняя на Kotlin 2.2.x (её метаданные
читает компилятор проекта 2.1.21). Новее (0.11+) собраны на Kotlin 2.3 и потребуют апгрейда Kotlin/Compose —
отдельной задачей.

Подробности и карта файлов — [`.claude/MCP.md`](.claude/MCP.md).

## День 17 — «Первый инструмент MCP»

**Задание дня:** реализовать свой MCP-сервер вокруг любого API; зарегистрировать инструмент с описанием
входных параметров и возвратом результата; подключить инструмент и **вызвать его из приложения**, получив
и использовав результат.

Здесь это не просто «дёрнуть API», а **самодостаточный умный MCP-сервис**: внутри сервера живёт
**агент-исследователь**, который сам ищет официальные источники, критикует найденное, докручивает поиск и
синтезирует ответ — полный цикл «рассуждение → поиск → парсинг → самокритика → синтез».

### Инструмент `get_visa_requirements(destination, citizenship, purpose)`

Актуальная визовая сводка для связки **гражданство → страна назначения**: нужна ли виза, куда подавать,
документы, сборы/сроки — **с цитатами на официальные источники и датой актуальности** (а не из «памяти» LLM).

### Агент-исследователь внутри MCP (агентный RAG с самокритикой)

```
get_visa_requirements(...)
  ├─ Поиск официальных источников: Tavily (осн.) → DuckDuckGo (фоллбэк) → Wikipedia (аварийно)
  │     официальные домены (gov / gouv / mfa / vfsglobal / посольства) — в приоритете
  ├─ Дотягивание полного текста доступных официальных страниц (VFS-инфо/gov часто открыты)
  ├─ Агент-критик (DeepSeek): что подтверждено официально? чего нет? «неудобные вопросы»
  │     (точно для этого гражданства? актуально? где офиц. список доков?) → новые запросы или done
  ├─ до 3 раундов докрутки
  └─ Агент-синтезатор (DeepSeek): структура + цитаты на офиц. URL + «Пробелы» + дата + дисклеймер
```

Почему так, а не «спарсить офиц. сайт напрямую»: порталы подачи (France-Visas, кабинет VFS) закрыты
Cloudflare/CAPTCHA — их не скрейпить. Поэтому ищем через поисковый индекс (он уже содержит контент
официальных страниц) и цитируем найденные официальные URL.

### Как подключено и вызывается из приложения

Сервер получает ключи (DeepSeek — для агентов, Tavily — для поиска) и регистрирует инструмент с input-схемой.
В окне **«Инструменты MCP»** появились поля **«Страна назначения»** и **«Гражданство»** + кнопка **«Узнать»** →
приложение через MCP вызывает `get_visa_requirements` и показывает структурную сводку. Консольная проверка —
`gradlew runMcpDemo`.

| Шаг задачи | Реализация |
|---|---|
| MCP-сервер вокруг API | поиск (Tavily/DDG) + Wikipedia, внутри сервера |
| Регистрация инструмента + input-схема | `get_visa_requirements(destination, citizenship, purpose)` |
| Возврат результата | структурная визовая сводка с цитатами и датой |
| Вызов из приложения + использование | поля + «Узнать» в окне «Инструменты MCP» → ответ в UI |

> **Фаза 2 (сделано):** автономный вызов инструмента **самим чат-агентом** — `tools`/`tool_calls` в
> tool-loop (`LlmClient`), инструменты на стадиях интервьюера и ассистента; в ленте видна строка
> `🔧 get_visa_requirements(…)`. Плюс правки логики: инжект текущей даты в промпт, честные статусы
> `[checklist]` (статус «загружен» ставит только система по реальным `docs`), чёткие правила
> `[SIMPLE]`/`[READY]`/`[ASK]` у интервьюера.

Подробности — [`.claude/MCP.md`](.claude/MCP.md).

## День 18 — «Планировщик и фоновые задачи»

Фоновый **периодический визовый дайджест**: MCP-сервер по расписанию собирает свежую сводку по
сохранённым странам (переиспользуя research-агента Дня 17), копит снимки в **SQLite** и по запросу
отдаёт **агрегированный** результат. Сервер крутится **24/7** на VPS — это и есть «агент, который
периодически выдаёт сводку».

| Критерий дня | Как закрыт |
|---|---|
| Хранение данных | **SQLite** (`mcp/scheduler/DigestStore.kt`): таблицы `digest_country` + `digest_snapshot` |
| Выполнение по расписанию | `mcp/scheduler/DigestScheduler.kt` — корутинный цикл, раз в `DIGEST_INTERVAL_MINUTES` |
| Агрегированный результат | тул `get_visa_digest` — по каждой стране свежая сводка + когда собрана + число снимков |
| Агент 24/7 | fat-jar на VPS под **systemd**, за **Caddy** (HTTPS), транспорт **SSE** + bearer-токен |

**Тулзы планировщика:** `add_digest_country(destination, citizenship?, purpose?)` (подписать + сразу
первый сбор), `list_digest_countries`, `remove_digest_country`, `get_visa_digest` (агрегат). Плюс
прежние `ping`, `get_visa_requirements`.

**Транспорт-агностичность.** Один и тот же сервер (`VisaMcpServer.kt`, `buildServer()`) поднимается в
двух режимах по env `MCP_TRANSPORT`: `stdio` (локально, подпроцесс приложения) или `sse` (Ktor
`embeddedServer` + `Application.mcp`, авторизация `Authentication { bearer }`). Десктоп-клиент
(`data/McpClient.kt`) умеет оба: подпроцесс (stdio) или удалённо по SSE (`sseUrl` + `authToken`).

**Деплой (remote, 24/7).** Сервер ставится на VPS как **systemd-сервис** (non-root пользователь,
`Restart=always`, БД в каталоге данных сервиса), за **reverse-proxy** с авто-HTTPS (напр. Caddy); секреты
(ключи + `MCP_AUTH_TOKEN`) — в env-файле на сервере, **не** в репозитории. Десктоп-агент ходит туда:
в `config.json` — `mcpRemoteUrl` + `mcpRemoteToken`; тогда `agentTools` создаётся как удалённый
`McpClient` и чат-агент сам зовёт дайджест.

```powershell
.\gradlew.bat mcpServerJar     # fat-jar → build\libs\visa-mcp-server-all.jar (java -jar … на VPS)
```

Env сервера: `MCP_TRANSPORT` (`stdio`|`sse`) · `MCP_HOST` (по умолч. `127.0.0.1` — за reverse-proxy) ·
`MCP_PORT` · `MCP_AUTH_TOKEN` · `DIGEST_DB` · `DIGEST_INTERVAL_MINUTES` · `DEEPSEEK_API_KEY` · `TAVILY_API_KEY`.

**Актуальные данные со ссылками на всех стадиях (доработка).** Синтез `get_visa_requirements` (свежие
требования + официальные источники) теперь собирается **один раз на кейс**, кладётся в досье задачи и
подмешивается блоком `[СПРАВКА]` во **все** стадии автомата — планирование, **выполнение**, проверку.
Исполнитель тоже получил инструменты и цитирует конкретные официальные `https://`-ссылки, а не отделывается
«уточните на сайте». Заодно: гражданство берётся из профиля, если пользователь не назвал его в диалоге
(иначе агент не доходил до плана), а серверный синтезатор отдаёт **полные** `https://`-URL с приоритетом
официальных доменов. **Портал подачи и английское название страны добываются динамически** (живой поиск
официального домена / LLM-нормализация), а захардкоженные списки стран — лишь аварийный фолбэк: это
снимает риск устаревания ссылок и не ограничивает сервис заранее прописанным набором стран.

Подробности — [`.claude/MCP.md`](.claude/MCP.md).

## День 19 — «Композиция MCP-инструментов»

**Задание дня:** создать несколько MCP-инструментов и собрать из них **пайплайн** (получить → обработать →
сохранить); проверить **автоматическое выполнение цепочки** и **корректность передачи данных** между
инструментами.

Три композируемых тула на нашем MCP-сервере (`mcp/VisaMcpServer.kt`):

| Тул | Роль | Что делает |
|---|---|---|
| `visa_search(query)` | 1 — получить | живой поиск официальных источников → СЫРОЙ список (заголовки, URL, сниппеты) |
| `visa_summarize(text, focus?)` | 2 — обработать | внутренним LLM сжимает присланный текст в выжимку, СОХРАНЯЯ `https://`-ссылки |
| `save_report(filename, content)` | 3 — сохранить | пишет контент в файл-отчёт на сервере, возвращает путь + размер + контент |

**Цепочка:** вывод каждого тула идёт на вход следующему — `visa_search` → `visa_summarize(вывод search)` →
`save_report(вывод summarize)`.

**Два режима запуска** (окно «Инструменты MCP» → секция пайплайна):
- **Агентный** — модели даём три тула и цель; tool-loop (`LlmClient`) САМ вызывает их по порядку (видно по
  `🔧`-трейсам) — настоящая авто-композиция MCP-инструментов.
- **Детерминированный** — код вызывает три тула по порядку, показывая вход/выход каждого шага (наглядная
  передача данных между инструментами).

Пайплайн-тулы НЕ отдаются основному агенту-консультанту (у него есть `get_visa_requirements`) — они только
для демонстрации композиции. Проверено end-to-end: оба режима выполняют цепочку, вывод корректно передаётся
между тулами, итоговый отчёт сохраняется в файл.

Подробности — [`.claude/MCP.md`](.claude/MCP.md).

## День 20 — «Skill + CLI как альтернатива MCP»

**Идея дня:** дать агенту второй механизм инструментов — **локальный Skill + CLI** — рядом с MCP, с
переключателями в UI и наглядным сравнением по токенам. Суть из лекции: схемы MCP грузятся в **каждый**
sampling call (idle-оверхед, рост на батче), а `SKILL.md` грузится **по требованию** (idle = 0). Вывод
«Цой жив»: **удалённое → MCP, локальное → Skill + CLI**.

**Что добавлено:**
- **`visa-cli`** — локальный CLI (gradle-задача `visaCliJar`): `docs` (приложенные документы аккаунта) и
  `prompt-tune` (collect / apply / list / reset). Запускается без сети.
- **`SkillRunner` (порт) + `CliSkillRunner`** — выполняет ТОЛЬКО `visa-cli`, через `ProcessBuilder` без
  shell (нейтрализация «логических бомб»). **`SkillEngine`** — текстовый skill-loop: модель пишет
  `[CLI] visa-cli …` → выполняем → `[CLI_RESULT] …` → повтор. `SKILL.md` грузится по требованию.
- **Панель «Коннекторы агента»** (иконка-ползунки в композере) — переключатели **MCP / Skill: документы /
  Skill: автоулучшение промтов**; вопрос «через MCP» vs «через Skill» с показом токенов и строкой
  «Δ prompt-токенов: MCP … vs Skill … — дороже в N×». Выключенный MCP реально отключается у агента.
- **Навык 1 — `docs`:** «какие документы я приложил?» → агент сам зовёт `visa-cli docs` и отвечает.
  (У удалённого MCP доступа к локальным файлам нет — это и показывает нишу Skill + CLI.) При приложении файла
  навык срабатывает сам. `visa-cli docs check` читает **текст PDF** (PDFBox) и агент **сверяет, что все
  документы оформлены на ОДНОГО заявителя** (ФИО/даты) — если файлы от разных людей, выводит ⚠️ и не
  принимает пакет. Сканы без текстового слоя помечаются «проверьте вручную».
- **Навык 2 — `prompt-tune` (самоулучшение промтов, человек в контуре):** анализирует диалоги, **предлагает**
  точечные добавки к промтам ролей под конкретного пользователя (с доказательствами-цитатами) и применяет
  их **только по подтверждению**. Перила: **A** только аддитивные добавки; **B** только с подтверждения;
  **C** whitelist «мягких» ролей (безопасность/отказы/стадии не тюнятся); **D** «Сбросить персонализацию».
  Добавки хранятся пер-аккаунт (`prompt-overrides.json`) и инжектятся в роли блоком `[ПЕРСОНАЛИЗАЦИЯ]`.

**Замер (тот же вопрос про локальные документы):** Skill + CLI — `prompt ≈ 1.1k` токенов и **отвечает**;
MCP — `prompt ≈ 1.7–4.9k` (8 схем в каждый запрос) и **не может** ответить (нет доступа к локальным файлам).

**Несколько MCP-серверов (оркестрация).** В задании дня — подключить ещё один MCP. Добавлен
**`McpRouter`** (тот же порт `ToolGateway`, внутри — несколько серверов): агент видит объединённый список
тулзов, а вызов маршрутизируется в сервер-владелец по имени инструмента (при коллизии — неймспейс
`<server>__<tool>`, в описании — метка сервера). Вторым сервером подключается **стороннее MCP**
`@modelcontextprotocol/server-everything` (npx, транспорт stdio) — тумблер «MCP — server-everything» в
панели коннекторов. Проверено: роутер видит **21 тул** (8 `[visa-info]` + 13 `[server-everything]`) и
корректно роутит вызов `echo` в сторонний сервер. В окне «Инструменты MCP» при включённом стороннем сервере
видна **разбивка по серверам** (сколько тулзов с каждого) и кнопка **«Тест стороннего MCP (echo)»** —
живая проверка, что второй сервер отвечает.

## Принятые решения

- **Compose for Desktop**, а не веб/Electron — тот же Jetpack Compose, нативное окно, упаковка в `.exe`
  с встроенным рантаймом (JVM ставить не нужно).
- **Clean Architecture** (`domain` без зависимостей · `data` · `ui`) + ручной DI (KISS, без фреймворка).
- **Память управляется автоматически, без пользовательских «режимов».** Пользователь не выбирает стратегию —
  агент сам решает, что положить в окно. Это убирает антипаттерн «вали всё в один промпт».
- **Авто-суммаризация при 30%** (агрессивно, но безопасно) — раннее сворачивание сырого хвоста не теряет
  смысл, т.к. факты сохранены в слоях памяти; токены и стоимость держатся стабильными.
- **Извлечение фактов всегда на `deepseek-chat`** — служебные вызовы должны быть стабильными и дешёвыми,
  независимо от выбранной в чате модели (бесплатные модели OpenRouter периодически лимитятся).
- **Долговременная память — в Markdown** (`profile.md`, `decisions.md`): человекочитаемо, можно править руками,
  легко грепать, переживает сессии. Рабочая память — структурно (JSON), её редко правят вручную.
- **Ключи — в `~/.adventai/config.json`** (или переменные окружения), не в репозитории.
- **Ошибки API не проглатываются** — показывается реальная причина (например, лимит 429), а не «пустой ответ».
- **UI в стиле Claude Code** — спокойный минимализм, выбор модели снизу, диалоги слева, скругления, один акцент.
