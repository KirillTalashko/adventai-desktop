# Контекстное окно и управление контекстом

## Принцип

Модель **не помнит ничего сама** — всё нужное передаётся в `messages` каждого запроса. Качество ответа =
качество переданного контекста. Управление контекстом — **под капотом, без пользовательских режимов**.

## Единый конвейер (`domain/Memory.kt → ContextAssembler`)

Каждый запрос собирается одинаково:

```
system( базовый промпт + [ДОЛГОВРЕМЕННАЯ] + [РАБОЧАЯ] [+ [РЕЗЮМЕ]] ) + последние N=12 реплик
```

- **Пока окно свободно** (заполнение < 30%): отправляется **вся история** — ничего не теряем.
- **При заполнении ≥ 30%**: старый хвост (всё старше последних N=12) сворачивается в **rolling summary**
  и подмешивается блоком `[РЕЗЮМЕ РАННЕЙ ЧАСТИ ДИАЛОГА]`; в `messages` идут только последние 12.

Смысл не теряется, потому что важное живёт в долговременной/рабочей памяти и в резюме (см. `MEMORY_MODEL.md`).

## Заполнение окна

- `contextFill` = `usage.prompt` последнего ответа ÷ `ModelOption.contextLimit` (`ui/ChatState.kt`).
- Лимиты в `data/Models.kt`: DeepSeek 65 536 · Llama/DeepSeek-V3 131 072 · Gemini 1 000 000.
- Порог авто-суммаризации — константа `ContextAssembler.SUMMARY_FILL = 0.30`.

## Где какая техника (всё под капотом, не режимы)

| Техника | Слой | Под капотом |
|---|---|---|
| **Sliding window** | краткосрочная | последние N=12 реплик дословно |
| **Sticky facts** | рабочая (+ промоушен в долговременную) | факты задачи «прибиты», не выпадают при обрезке (сейчас — рабочая память вручную; авто-извлечение — TODO) |
| **Summary** | переполнение | свёртка старого хвоста при ≥30% |
| **Branching** | не про окно | отдельная фича: форк диалога для альтернатив, у каждой ветки своё чистое окно (TODO) |

## Токены

- `usage` (prompt/completion/total) сохраняется в каждом ответе (`Message.usage`).
- UI: чип под ответом + сумма по сессии в композере (+ стоимость через `ModelOption.costUsd`).
- Суммаризация работает **тихо** (без баннера). Бесплатные модели OpenRouter лимитятся (429) — клиент показывает причину.
