## 0. TL;DR недели

- **MCP (Model Context Protocol)** — открытый **стандарт** (Anthropic), не фреймворк. Единый слой между LLM и
  внешними сервисами: таск-трекеры, базы знаний, CI/CD, 1С/SAP, календарь/почта. Реализаций много (Python, **Kotlin**, …).
- **MCP ≠ API.** У каждого сервиса свой API без единых правил; MCP сводит различия апишек и различия нейронок к типовому протоколу (JSON-RPC, `tools/list` → схемы инструментов).
- **Главная мысль лекции — экономика токенов.** Схемы MCP-тулзов **грузятся в КАЖДЫЙ sampling call** (idle-overhead даже на «сколько времени?»), а при батчинге расход растёт **квадратично**. **Skill + CLI** грузится **по требованию** (idle = 0) и растёт линейно → дешевле.
- **Провокация «MCP deprecated».** На практике — **«Цой жив»**: skill+cli требует локального CLI-инструмента; для удалённых сервисов MCP остаётся нужен. Вывод: уметь **и то, и то**, знать когда что дешевле.
- **Наша задача недели:** дать агенту «Визовый специалист» **инструменты** (MCP-клиент в своём агенте, не в готовом код-ассистенте), затем **опубликовать свой MCP-сервер** на данные приложения, потом сделать **Skill + CLI** альтернативу и **измерить** разницу в токенах нашим же счётчиком.

## 1. Конспект лекции (что усвоить перед кодом)

| Тема | Суть |
|---|---|
| **Зачем** | Корпоративные системы/знания/инфраструктура недоступны «общей» нейронке → нужен стандартный доступ. |
| **Стандарт, не фреймворк** | Путаница из-за множества реализаций на разных языках. MCP — это протокол. |
| **Куда подключать** | Готовые агенты (Claude, ChatGPT) **или свой агент**. Любая LLM умеет общаться с MCP. |
| **MCP ≠ API** | Каждый сервис парсится по-своему; MCP — общий протокол поверх различий. Соблюдают, т.к. сразу попадаешь во все нейронки. |
| **Плюсы** | Доступ из AI без кода · **изолированный доступ** (даёшь только нужные `tools`) · единый стандарт. |
| **Минусы** | **Безопасность** — «логические бомбы» (в названии встречи прилетает `rm -rf` → сервер выполнил). Нужно писать MCP-сервер (не всегда оправдано — для своего микросервиса проще API). |
| **Технически** | JSON-RPC `tools/list` → массив `tools` со схемами (`inputSchema`, `required`). Transports: **stdio** (локально) / HTTP-SSE (remote). |
| **Kotlin SDK** | `io.modelcontextprotocol:kotlin-sdk`. Сервер: `Server(...)` + `ServerOptions(...)`, `StdioServerTransport`, `server.addTool(name, description, inputSchema){ … }` → `CallToolResult`/`TextContent`. Ключевой слайд — **маппинг тула на конкретный HTTP-запрос**. |
| **Remote MCP** | VPS/публичный хост → серверное приложение → поверх него протокол MCP → публикация (HTTPS/домен) → добавить в custom tools агента. |
| **MCP token flow** | Стадии: user input → context collection (**get tools mcp**) → Sampling Call #1 (включает схему тулзов) → tool calling (JSON-RPC) → Sampling Call #2 (включает токены #1). |
| **Расчёт (пример)** | system 1500 + user 200 + (6 тулзов × 900) 5400 = **7100 input**; tool result 3500; response 600. На 2 sampling call ≈ **18 540 токенов**, полезного ввода ~2% — **~98% на обслуживание MCP**. |
| **MCP vs Skill (его репо)** | 46 тулзов → схемы ~5787 ток., **грузятся ВСЕГДА**. Skill+CLI: `SKILL.md` ~3760 + refs ~763 + bash ~200 = **~4723**, **по требованию**, **idle = 0**. |
| **3 источника overhead** | (1) **idle** (MCP грузится всегда) · (2) **batching** (MCP растёт ~квадратично; на 20 экшенов — космос) · (3) **schema weight** (тяжёлый тулинг). |
| **Вывод** | Skill+CLI — будущее по токенам, но MCP не умер: нужен для удалённых сервисов. Знать оба и когда что применять. |

**Цели недели (со слайда «Цели»):** ① познакомиться с MCP · ② реализовать для **публичных** тулзов (GitHub, Яндекс Трекер…) · ③ научиться **оркестрировать** MCP в своём агенте · ④ применить в реальной работе · ⑤ **сравнить с Skill + CLI** по токенам.

## 3. Архитектурные изменения (где в коде)

Опираемся на текущую карту (`.claude/ARCHITECTURE.md`): `domain/Ports.kt`, `domain/VisaAgent.kt`, `data/LlmClient.kt`.

### Новый доменный контракт инструментов
- **`domain/Tools.kt`** (новое): `Tool(name, description, inputSchema)`, `ToolCall(id, name, argsJson)`, `ToolResult(callId, content, isError)`.
- **`domain/Ports.kt`** (правка): новый порт
  - `ToolGateway` — `suspend fun listTools(): List<Tool>` + `suspend fun call(call: ToolCall): ToolResult`.
  - `LlmGateway.complete(messages, tools: List<Tool> = emptyList())` — пробрасываем схемы; `GatewayResponse` получает `toolCalls: List<ToolCall>` (если модель попросила инструмент).
- **`domain/VisaAgent.kt`** (правка) — **tool-loop**: пока `GatewayResponse.toolCalls` непуст → вызвать `ToolGateway.call(...)` для каждого → дописать результаты в сообщения → повторный `complete`. Лимит итераций (защита от зацикливания).

### Реализация в `data`
- **`data/McpClient.kt`** (новое) — реализует `ToolGateway` через **Kotlin MCP SDK** (`io.modelcontextprotocol:kotlin-sdk`). Клиентская сторона: запуск/подключение к MCP-серверам (stdio для локальных `npx`-серверов, HTTP-SSE для remote). Маппинг `Tool` ↔ MCP tool schema; `ToolCall` ↔ JSON-RPC `tools/call`.
- **`data/LlmClient.kt`** (правка) — добавить в chat-completions поле `tools` (OpenAI-совместимый формат; DeepSeek его поддерживает) и распарсить `tool_calls` из ответа. MCP `inputSchema` → OpenAI `function.parameters`.
- **`data/McpRegistry.kt`** (новое, опц.) — конфиг подключённых MCP-серверов (по аналогии с `claude_desktop_config.json` → блок `mcpServers`), хранение в `~/.adventai/mcp.json`.

### Зависимости (`build.gradle.kts`, со слайда «Зависимости/SDK»)

Деструктивные тулзы — read-only по умолчанию, мутации — за явным подтверждением (как пошаговое подтверждение в state-машине).

### День 17 — Tool-loop + оркестрация в `VisaAgent`
**Цель ③.** Полный цикл и изоляция доступа.

- **Изолированный доступ** как в лекции: отдаём только нужные `tools`.
- **Безопасность:** `InvariantGuard` фильтрует tool-calls (защита от «логических бомб»).
- **UI (стиль Claude Code):** показ вызова инструмента строкой `● tool(args)` в ленте + статус в нижней строке.
- **Приёмка:** задача с 2–3 tool-call отрабатывает за один ход; опасный аргумент блокируется стражем; видно в JSON.

### День 18 — Свой MCP-сервер (тулзы + транспорт-агностичность)
**Цель ② (своё).** Пишем сервер один раз — оба транспорта на общей логике.
- MCP-**сервер** на Kotlin SDK (`Server` + `ServerOptions`), экспортирует несколько **read-only** тулзов:
  - **`get_visa_news(country)`** — наш ключевой доменный тул: server-side ходит на **официальные сайты стран**, парсит свежие визовые новости/изменения, возвращает структурированный результат (тул → HTTP-запрос, как слайд 15 лекции);
  - `get_visa_checklist`, `list_docs` (DocStore), `get_decisions` (долговременная память аккаунта).
- **Два транспорта на общей логике тулзов:** `StdioServerTransport` (быстрая локальная отладка) и **SSE** (Ktor-эндпоинт `/sse` — для VPS на Дне 19). Флаг выбора транспорта.
- Точка входа — отдельный gradle-таск/`Main`-вариант (как `weather`-jar в лекции).
- **Безопасность:** только чтение, без выполнения команд; секреты/`config.json` наружу не отдаём.
- **Приёмка:** **MCP Inspector** (`npx @modelcontextprotocol/inspector`) и наш `McpClient` видят tools и получают чек-лист/решения **локально по обоим транспортам**; компиляция зелёная.

### День 19 — VPS: деплой remote MCP-сервера (HTTP-SSE)
**Цель ④ + развилка 3 (remote).** Поднимаем «настоящий» удалённый сервер (у пользователя VPS впервые — идём по шагам).
- Провижн VPS (Ubuntu), домен, **HTTPS** (Caddy с авто-TLS Let's Encrypt) поверх SSE-эндпоинта; токен-авторизация.
- Деплой jar (systemd-сервис) + reverse-proxy на `https://mcp.<домен>/sse`.
- **Наш собственный агент** (`McpClient`) подключается к remote-серверу по URL+токену — пока **только он** (публичный коннектор — позже).
- **Безопасность:** read-only, токен обязателен, ничего лишнего наружу; rate-limit.
- **Приёмка:** агент в десктоп-приложении получает данные **с VPS** (видно в `conversations/*.json`); MCP Inspector подключается к публичному URL. Заложен «выключатель» в публичный коннектор Claude.ai на потом.

## 5. Безопасность — «логические бомбы» (минусы из лекции)

- **Никакого произвольного bash/exec через MCP-тулзы.** Деструктивные операции — только за явным подтверждением пользователя (как пошаговое подтверждение state-машины).
- **`InvariantGuard` проверяет аргументы tool-call**, а не только текст: имя встречи/заметки/документа не должно приводить к команде (`rm -rf`, инъекции). Перегенерация в отказ при конфликте.
- **Изоляция доступа:** по умолчанию **read-only**; набор тулзов сужается по стадии задачи. Внешний MCP-сервер (Дни 18–19) — только чтение.
- **Свои секреты не утекают:** MCP-сервер не отдаёт `config.json`/ключи; данные аккаунтов изолированы (как в Дне 12).
- **Remote на VPS публично доступен (День 19)** → **токен-авторизация обязательна**, только read-only тулзы, никаких мутаций/записи, rate-limit на эндпоинт. Пока коннектор не публичный — ходит только наш агент с токеном.
