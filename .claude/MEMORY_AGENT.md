# Агент памяти (авто-наполнение)

Рабочая и долговременная память **наполняются сами** из диалога отдельным фоновым агентом — не вручную.

## Идея

Главный агент (визовый специалист) отвечает пользователю. Параллельно **отдельный агент памяти**
(`domain/MemoryExtractor.kt`) читает последние реплики, извлекает новые факты и раскладывает по слоям.
Это паттерн *memory extraction / reflection*.

## Как работает

```
ход: user → агент отвечает → (в фоне, не блокируя UI):
   MemoryExtractor.extract(последние 4 реплики + текущая память)
      → секции: ЦЕЛЬ / ОГРАНИЧЕНИЯ / ПРОФИЛЬ / РЕШЕНИЯ (только НОВОЕ)
      → маршрутизация: ЦЕЛЬ+ОГРАНИЧЕНИЯ → рабочая, ПРОФИЛЬ+РЕШЕНИЯ → долговременная
      → запись с дедупом в MemoryStore
```

- **Отдельная модель:** всегда **`deepseek-chat`** (стабильность), независимо от выбранной в чате. Если нет
  ключа DeepSeek — авто-наполнение просто не работает (ручное добавление в диалоге «Память» остаётся).
- **В фоне:** `ChatState.runExtraction` запускается из `onSuccess` через `scope.launch` — пользователь не ждёт.
- **Окно извлечения:** последние `EXTRACT_WINDOW = 4` сообщения (≈2 обмена) + снимок текущей памяти, чтобы
  модель возвращала только новое.
- **Без выдумок:** low-context, инструкция «строго из текста»; ошибка/пустой ответ → память не растёт.
- **Дедуп:** `FileMemoryStore` не добавляет уже известные факты (сравнение без регистра).

## Хранение

- **Долговременная** (per-агент) — markdown, человекочитаемо/редактируемо: `memory/profile.md` + `memory/decisions.md`.
- **Рабочая** (per-диалог) — структурно: `working/<id>.json` (goal, constraints).

## UI

Кнопка **«Память»** (`ui/Dialogs.kt → MemoryDialog`) показывает текущее содержимое обоих слоёв (видно, как
растёт по ходу диалога) и позволяет дополнять вручную. Авто-записи и ручные правки складываются в одни и те же файлы.
