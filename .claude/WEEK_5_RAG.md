## 0. TL;DR недели

- **RAG = Retrieval-Augmented Generation.** Паттерн: во **время инференса** дополняем ответ LLM
  **внешними/внутренними знаниями**, которых не было в тренировочной выборке модели.
- **RAG ≠ MCP.** MCP — общение с **внешними** сервисами (дёргаем сторонний tool). RAG — работа с
  **нашими собственными знаниями**: текст → вектор → сохранили в БД → на запрос достаём ближайшие по
  смыслу куски и подмешиваем в prompt. Никакой внешний tool при этом не дёргается.
- **Зачем:** «голая» LLM обучена на общем шуме (гитхаб/реддит/твиттер), про **наши** данные не знает →
  4 боли: **Context** (контекст неполный/смешанный/большой), **Performance** (перелопачивать всю базу
  долго), **Incorrect data** (вернёт не ту версию — у каждой команды/года свой процесс), **Lack of
  verified data** (нет источника → галлюцинация; хуже всего — *правдоподобная* галлюцинация).
- **Ключевой новый инструмент — embedding-модель** (не генеративная): текст → **вектор** (768/1024/3072
  чисел). Похожесть по смыслу = **косинусное сходство** (1 — синонимы, 0 — про разное, −1 —
  противоположность). Для проекта берём **Ollama + `nomic-embed-text`** (локально, бесплатно, 768).
- **Пайплайн RAG:** запрос → embed запроса → поиск ближайших чанков в векторной БД → combine с prompt →
  ответ LLM **со ссылкой на источник**. Плюс тюнинг: **chunking + overlap**, **normalization**,
  **reranking** (cross-encoder поверх top-20 bi-encoder).
- **Задача недели (со слайда «Цели»):** ① научиться работать с **эмбеддерами** · ② реализовать **поиск
  по базе знаний** · ③ **включить это в работу агента** (+ подключать **ссылки** на источники) ·
  ④ применить в реальной работе. **Отдельное задание недели — сделать reranking.**

## 1. Конспект лекции (что усвоить перед кодом)

| Тема | Суть |
|---|---|
| **Что такое RAG** | Дополнение LLM-ответа внешними знаниями **во время инференса**. Ответ = общая база модели + наши специфические знания. |
| **RAG vs MCP** | MCP = общение с **внешними источниками** (сторонний сервер, tool-call). RAG = общение с **внутренними знаниями** (векторная БД, подгрузка в контекст, без tool-call). |
| **Боли до RAG (LIMITS)** | **Context** · **Performance** · **Incorrect data response** · **Lack of verified data**. MD-файл в промпте не спасает: поиск затруднён, контекст неполный/смешанный/большой. |
| **Худшая боль** | Не просто неправильный ответ, а **правдоподобно-неправильный** (ложноположительный): не хватает компетенции/времени отличить — принимаешь ложь за правду. |
| **Матрица оптимизации LLM** | Оси: **Context optimization** («что модель должна знать») × **LLM optimization** («как должна действовать»). Квадранты: низ-лево **Prompt engineering**, верх-лево **RAG**, низ-право **Fine-tuning**, верх-право **All of the above**. |
| **Аддитивность** | Prompt / RAG / Fine-tuning — **additive, не exclusive**. В реале используют всё вместе; для конкретного кейса стекуются. |
| **Prompt-оптимизация** | Контекстная оптимизация в широком смысле (skills, субагенты, MD-файлы). Быстро/легко, но не структурировано и подвержено галлюцинациям. |
| **Fine-tuning** | LLM-оптимизация = меняем **веса**: модель иначе взвешивает уже имеющиеся знания. |
| **Пример без RAG** | «Какая у нас политика код-ревью?» → LLM из общего контекста: «обычно 1 approval, в течение недели» → **⚠ No source — hallucinated**. |
| **Пример c RAG** | Тот же вопрос → «минимум 2 approval до мержа, макс. время ревью 24 часа» + **📎 источник** `confluence/engineering/code-review-policy.md`. |
| **Где применять (WHERE)** | Корп. базы знаний (чаты/вики) · закрытые данные (не «светить») · спец-агенты (узкие ниши: медицина/физика) · глобальный анализ трендов · **цифровой двойник** (переписки/звонки/видео → персональная история). |
| **Цели недели** | Работать с эмбеддерами · поиск по базе знаний · включить в агента + **ссылки** · применить в реальной работе. |

## 2. Как это работает (пайплайн)

**Flowchart (слайд 19) — какой инструмент выбрать:**
```
Start → Prompt engineering → Evaluate and identify the gaps
        ├─ In-context memory  (нужны факты для ответа)      → RAG (Retrieval-augmented generation)
        └─ Learned memory     (нужен свой стиль/структура)  → Fine-tuning
   Примечание: additive, not exclusive — для ряда кейсов стекуются.
```

**Pipeline diagram (слайд 20, пример «What is the population of Canada?»):**
```
User → Chat Completion → Question
                         → (нет ответа в своей базе) → Documents → Embeddings → Retrieval
                         → Combine content with prompt
                         → Content: «Population of Canada 39,566,248 as of Oct 19, 2023 (UN data)»
                         → Answer со ссылкой на источник
```
> По духу похоже на MCP, но **различие**: RAG крутится на **embedding-моделях** и **своей векторной БД**,
> а не на tool-call к чужому сервису.

## 3. Модели: генеративная vs embedding

| | **Generative model** | **Embedding model** |
|---|---|---|
| Делает | генерирует новый текст | кодирует текст в **вектор** |
| Учится | понимать контекст, строить фразы | понимать **семантические связи** |
| Вход→выход | текст → текст | текст → **вектор** |
| Для чего | ответы на вопросы, код-ассистенты | **поиск похожих текстов** |
| Примеры | ChatGPT, Claude AI, YandexGPT, LLama3 | **Ollama**, OpenAI API, Voyage AI API, Jina.ai |

**Вектор** = «много циферок» (`"кот" → [0.1, -0.34, 0.88, …]` — например, 768 чисел). Похожесть =
**косинусное сходство**: `cos=1` синонимы/близко, `cos=0` про разное (не пересекаются), `cos=-1`
противоположно (красный/синий). Свои фразы тоже векторизуем и кладём рядом с готовыми.

**Модели эмбеддингов (слайд 26):**

| Модель | Среда | Размер вектора | Замечание |
|---|---|---|---|
| **nomic-embed-text** | **Local (Ollama)** | **768** | Fast, free ← наш выбор |
| text-embedding-3-large | OpenAI API | 3072 | Very accurate |
| voyage-large-3 | Voyage AI API | 1536 | Top-tier |
| jina-embeddings-v3 | Jina.ai | 1024 | Multilanguage |
| gte-base | Hugging Face | 768 | Fast, free |

**Как embedding-модель считает вектор (слайды 28–35, пример «SpaceX was founded in 2002»):**
```
Input Text → Tokenization (["Space","X"," was"," founded"," in"," 2002"])
           → Embeddings Layer (каждый токен → вектор)
           → Transformer Layer (self-attention: Attention(Q,K,V)=softmax(QKᵀ/√dk)·V)
           → Pooling («pulling»: усреднение векторов токенов = нормализованный вектор всего предложения)
           → Vector
```
Обучение embedding-модели: на **размеченных** тройках (позитивный/негативный пример) минимизируется
контрастив-лосс `Loss = max(0, margin + cos(neg,A) − cos(A,pos))` — притягиваем близкие по смыслу,
отталкиваем далёкие. **Нам это делать не нужно** — берём готовую модель, она уже умеет считать вектора.

## 4. Chunking (нарезка) + overlap

- **Fixed-size** (тупо каждые N токенов) — **⚠ рвёт предложения, контекст теряется**.
- **Semantic** (по смысловым блокам) — **✓ каждый чанк = законченная мысль**.
- **Overlap** — соседние чанки **перекрываются**, чтобы границы «были на замке»:
  ```
  chunk_size = 500   overlap = 50
  Chunk 1: tokens 1–500
  Chunk 2: tokens 451–950
  Chunk 3: tokens 901–1400
  ```
  Без overlap — потеря контекста на стыках. С overlap — граница защищена.
- **Размер чанка и overlap подбираются экспериментально** под конкретные данные/задачу (в RAG «без теории
  не обойдёшься» — много эмпирики).

## 5. Почему Ollama (слайд 40)

- **Безопасность** — локально, данные не покидают компьютер.
- **Нулевая цена** — не нужен API-ключ и оплата за токены.
- **Совместимость** — легко потом перейти на облачный эмбеддер.
- **Тестирование** — если RAG хорошо работает на малой модели, на большой будет ещё лучше.
- Скорость инференса эмбеддера не критична (векторизуем/девекторизуем нечасто); параметров мало
  (768/3072, а не миллиарды).

```bash

ollama --version               # check
ollama serve &                 # launch
ollama pull nomic-embed-text   # загрузить модель эмбеддингов
```

**Kotlin-клиент (embeddings по HTTP на localhost):**
```kotlin
// Models
@Serializable
data class OllamaEmbeddingsRequest(
    val model: String = "nomic-embed-text",
    val prompt: String,
)

@Serializable
data class OllamaEmbeddingsResponse(
    val embedding: List<Double>,
)

// клиент: POST на http://localhost:11434/api/embeddings (обычный HTTP-клиент),
// suspend fun embed(text): FloatArray  → response.embedding.toFloatArray()
```
> Под капотом — **обычный HTTP-клиент**, только `localhost` (локальная модель). Если вынести Ollama на
> VPS — просто меняем `localhost` на адрес VPS, логика та же. Возвращаются близкие/далёкие чанки → их
> подмешиваем в response LLM и показываем человеку (со ссылкой), либо гоняем ещё круг (reranking/tuning).

## 7. Tuning качества RAG (слайды 50–51)

- **Chunks** — чанки 500–1000 токенов (экспериментируй).
- **Overlapping** — перекрытие 50–100 токенов между чанками.
- **Storage** — вектора сохранять в БД (векторное хранилище).
- **Normalization** — привести вектор к диапазону **[0, 1]** (просто: делим все числа на максимум вектора).

## 8. Reranking (слайды 53–54) — **отдельное задание недели**

**Проблема:** bi-encoder + cosine отдаёт **Top-20** быстро, но неточно. Пример запроса
«Как настроить CI/CD для Kotlin Multiplatform?»:
- ✓ «Настройка GitHub Actions для KMP» — релевантно
- ✗ «Обзор CI/CD инструментов 2023» — похоже по вектору (есть «CI/CD»), но на вопрос не отвечает

**Решение:** добавить cross-encoder для переоценки:
```
Query → Bi-encoder (Top-20) → Cross-encoder (Rerank → Top-5) → LLM → ✓ точный ответ
```

| | **Bi-encoder** | **Cross-encoder** |
|---|---|---|
| Как работает | отдельные эмбеддинги (query и doc порознь) | **Query + Doc вместе** |
| Скорость | быстро (миллисекунды) | медленно (секунды) |
| Точность | средняя | **высокая** |
| Когда | 1-й этап — **отсев** (широкий) | 2-й этап — **точный отбор** |

## 9. Метрики качества RAG (слайд 55)

| Метрика | Вопрос | Что меряет | Низкое значение → |
|---|---|---|---|
| **Context Relevance** | Нашли ли мы правильные чанки? | долю релевантных документов в выдаче поиска | плохой retrieval |
| **Faithfulness** | Ответ основан на контексте? | не галлюцинирует ли модель (всё из чанков) | галлюцинации |
| **Answer Correctness** | Финальный ответ правильный? | сравнение с эталонным ответом (итоговая метрика) | всё плохо |

**Практика:** в конце каждого ответа — мини-анкета (👍/👎: правильные ли чанки / был ли ответ по контексту
со ссылками / верен ли финальный ответ). Можно **автоматизировать**: набор эталонных вопросов+ответов,
прогонять другими моделями и мерить точность.

## 10. Пределы RAG (слайд 56) — когда НЕ поможет и что вместо

| RAG **не поможет** | Что делать вместо |
|---|---|
| **Нет данных** (мало / плохо размечены) | **Fine Tuning** (обучать свою модель) |
| **Нужна логика** (сложные правила, не просто достать факт) | **Prompt Engineering** |
| **Нужен стиль** (вежливость/спор/тон — RAG хранит сырые вектора для косинус-поиска) | **Agents + Tools** (тема персонализации) |
| **Устаревший формат** (модель не поддерживает — амбарные/бухгалтерские книги и т.п.) | **Multimodal RAG** / оцифровка |

## 11. Что это значит для «Визового специалиста» (наш проект)

RAG ложится на агента естественно — у нас уже есть локальные знания и требование ссылок:

- **База знаний = визовые данные.** Сырьё в `/data` (или `~/.adventai/…`): визовые требования, чек-листы,
  памятки по странам, консультации. Нарезаем на чанки (semantic + overlap), эмбеддим `nomic-embed-text`
  через **локальный Ollama**, храним вектора в БД.
- **Поиск по базе знаний** на стадиях `INTAKE`/`ASSIST`/`EXECUTION`: запрос пользователя → embed → top-k
  ближайших чанков → подмешать в prompt роли **со ссылкой на источник**. Это прямо усиливает нашу планку
  качества `[[mcp-answer-quality-bar]]` («ответ со ссылками + свежие данные на каждой стадии»).
- **RAG vs существующий `get_visa_requirements` (MCP/tool).** MCP-инструмент = внешние/удалённые данные;
  RAG = наши внутренние знания. Аддитивны: справка от инструмента + RAG по локальной базе → в один prompt.
- **Reranking (задание недели):** bi-encoder top-20 по нашей базе → cross-encoder rerank → top-5 в LLM.
- **Ссылки — обязательны.** Как «собери ссылки по вопросу», но только **по нашей базе знаний** — это и есть
  цель недели «включить в работу агента + подключать ссылки».
- **Стек совпадает:** Kotlin/JVM + Ktor HTTP-клиент к `localhost:11434` — переиспользуем паттерн `LlmClient`.
